System Design

Three-Layer Architecture

EchoEats follows a clean separation of concerns: a lightweight frontend, a stateless REST API, and a relational database — each independently deployable.

Layer 1 · Client
Frontend
Static HTML/CSS/JS served as a PWA. No build step required — all pages link to a shared stylesheet and a single main.js. Designed for zero-dependency portability.
HTML5 CSS3 Custom Props Vanilla JS PWA localStorage
Layer 2 · Server
Backend API
A stateless REST API built with Node.js and Express. Handles auth (JWT), listing CRUD, dynamic pricing recalculation (cron every 15 min), and Stripe webhook processing.
Node.js Express JWT Auth node-cron Stripe SDK
Layer 3 · Data
Database
PostgreSQL stores users, listings, reservations, transactions, and impact stats. Row-level locking prevents double-reservations. Redis caches hot listing prices between cron runs.
PostgreSQL 16 Redis Prisma ORM Row Locking

Layer 1

Frontend Technologies

Intentionally lightweight — no framework overhead, instant load times, and full team accessibility regardless of JS framework experience.

📄
HTML5 + CSS3
Markup & Styling
Semantic HTML5 structure across 10 pages, driven by a single shared stylesheet using CSS custom properties for the design token system (colors, radii, shadows).
Why: Zero build step. Every team member can open and edit any file instantly. CSS variables give us a full design system without Tailwind or Sass.
⚡
Vanilla JavaScript
Interactivity & Logic
All client-side logic — auth flows, dynamic pricing simulation, toast notifications, localStorage state, drag-and-drop task board — in a single main.js plus page-level scripts.
Why: No bundle size, no dependency risk. The dynamic pricing ticker, reserve button, and scroll animations all run in under 5KB of JS.
📱
Progressive Web App
PWA + Offline Support
A service worker caches the shell, fonts, and last-viewed listings. Users can add EchoEats to their home screen and browse previously-seen deals offline.
Why: Critical for Bahrain's mobile-first user base. PWA gives native-app feel without App Store friction or React Native complexity.
🗂
localStorage
Client-Side State (Demo)
All demo auth, user profiles, listings, and task board state is persisted in localStorage. This powers a fully functional prototype with zero backend dependency for grading/demo purposes.
Why: Lets the entire frontend be evaluated standalone. In production, this is replaced by API calls to the Node backend.
🅰
Plus Jakarta Sans
Typography
Primary sans-serif via Google Fonts. Paired with Instrument Serif for editorial headings. Loaded with display=swap to prevent layout shift during font loading.
Why: Modern geometric humanist — legible at small sizes on mobile, distinctive at large display sizes. Free and widely supported.
🎞
IntersectionObserver
Scroll Animations
Native browser API drives all .reveal scroll animations — no GSAP, no ScrollMagic. Elements fade and slide in as they enter the viewport with staggered delays.
Why: Zero KB of animation library. Performant, GPU-composited transforms only. Works perfectly across all modern browsers.

Layer 2

Backend Technologies

A stateless REST API designed around the core pricing engine — with scheduled jobs, webhook handling, and role-based access control baked in.

🟢
Node.js 20 LTS
Runtime Environment
JavaScript runtime for the API server. Uses the LTS release for stability. The pricing engine's decay calculations run in the same language as the frontend, enabling shared utility functions.
Why: Full-stack JavaScript means the team works in one language. Event-loop I/O handles concurrent reservation requests efficiently without thread management.
🚂
Express.js 4
Web Framework
Minimal, unopinionated REST framework. Routes are split by domain: /api/auth, /api/listings, /api/reservations, /api/vendors, /api/impact. Middleware handles CORS, rate-limiting, and validation.
Why: Lightweight and well-understood by the team. We don't need the full NestJS/Fastify ecosystem — Express gives us exactly what's required.
🔐
JWT + bcrypt
Authentication
JSON Web Tokens for stateless auth. Consumer and vendor roles encoded in the token payload. Passwords hashed with bcrypt (12 rounds). Refresh token rotation via httpOnly cookie.
Why: Stateless tokens scale horizontally without session store coordination. Role-based middleware protects vendor-only and consumer-only routes cleanly.
⏱
node-cron
Pricing Engine Scheduler
Scheduled job runs every 15 minutes to recalculate all active listing prices using the chosen decay curve (linear, exponential, or stepped). Expired listings are auto-archived.
Why: The 15-minute tick is central to EchoEats UX. node-cron keeps this logic server-side so prices are consistent across all clients simultaneously.
🧪
Zod
Input Validation
Schema-based validation for all incoming API request bodies. Listing creation, reservation requests, and user registration are all validated before hitting the database.
Why: TypeScript-first with great error messages. Prevents invalid data from reaching Prisma and the DB — critical for price floor / expiry validation.
📬
Nodemailer
Transactional Email
Sends pickup confirmations, reservation hold warnings (at 10 min), and vendor payout summaries. Configured with SendGrid as the SMTP relay in production.
Why: Simple, battle-tested, zero vendor lock-in for the SMTP provider. The 15-minute reservation window makes timely email notifications essential.

Layer 3

Data Layer

Relational data with strong consistency guarantees — critical for the reservation lock system that prevents two users buying the same item.

🐘
PostgreSQL 16
Primary Database
Stores users, listings, reservations, transactions, badges, and impact stats. Row-level locking (SELECT FOR UPDATE) ensures atomic reservation — no double-booking even under concurrent load.
Why: ACID guarantees are non-negotiable for payments and reservations. Postgres's JSONB columns also store the flexible decay curve config per listing without a separate table.
🔷
Prisma ORM
Database Access Layer
Type-safe database client generated from a schema.prisma file. Handles migrations, relationships, and query building. Used for all CRUD operations on listings, users, and reservations.
Why: Auto-generated types catch data shape errors at compile time. Migrations-as-code means schema changes are version-controlled alongside application code.
🔴
Redis 7
Cache + Rate Limiter
Caches the current calculated price of every active listing between cron ticks. Also powers the API rate limiter (100 req/min per IP) and the 15-minute reservation TTL lock.
Why: Price reads are extremely frequent (every user pageload). Redis serves them in <1ms vs ~5ms Postgres. The TTL-based reservation lock is a natural Redis pattern.

Monetisation

Payments & Payouts

End-to-end payment infrastructure — from consumer checkout to automatic vendor payout — without EchoEats ever touching card data.

💳
Stripe Checkout
Consumer Payments
Hosted checkout handles card processing, 3DS, Apple Pay, and Google Pay. A PaymentIntent is created server-side at the current dynamic price, then confirmed in Stripe's hosted UI.
Why: PCI-compliant out of the box. Stripe's hosted checkout means zero card data touches EchoEats servers — critical for a student project with real payment flows.
🏦
Stripe Connect
Vendor Payouts
Each vendor onboards via Stripe Connect Express. When a sale completes, the platform fee (15%) is split automatically — the vendor receives their portion instantly via Stripe's managed payouts.
Why: Removes all payout complexity. No manual transfers, no invoicing, no currency risk. Stripe handles KYC for vendors in GCC markets including Bahrain.
🪝
Stripe Webhooks
Event Processing
Listens for payment_intent.succeeded, payment_intent.payment_failed, and checkout.session.completed events to confirm reservations, update listing status, and trigger impact stat updates.
Why: Asynchronous — the frontend doesn't wait for the DB write. The webhook is the source of truth for reservation confirmation, handling network drops gracefully.

Infrastructure

DevOps & Deployment

CI/CD pipeline, hosting, and monitoring — keeping EchoEats live and observable at all times.

🐙
GitHub Actions
CI/CD Pipeline
On every push to main: run ESLint, Jest unit tests, Prisma migration check, then deploy. PRs require passing checks and one peer review before merge.
Why: Native to GitHub where our repo lives. Free for the team. Enforces code quality gates without relying on individual discipline.
🔺
Vercel + Railway
Hosting
Static frontend deployed to Vercel (edge CDN, global). Node.js API on Railway with auto-scaling. PostgreSQL and Redis also hosted on Railway in the same private network.
Why: Zero-config deploys from GitHub. Free tier covers our demo traffic. Same-network DB colocation keeps API latency under 2ms for Postgres queries.
🧪
Jest + Supertest
Testing
Unit tests for the pricing engine's decay curve functions (linear, exponential, stepped). Integration tests for reservation endpoints using Supertest against a test database.
Why: The pricing math is the core of EchoEats — it needs bullet-proof tests. Supertest lets us test the full Express middleware stack without spinning up a server.

Decision Rationale

Why this stack wins

Four key reasons this stack outperforms alternatives for our specific project constraints.

🎓
Fits the academic context
The project is graded and demoed by opening HTML files — no npm install, no localhost, no build errors mid-demo. Our stack works identically on any machine with a browser.
⚡
Speed to working prototype
We shipped a 10-page interactive prototype in two sprints. No time was lost on framework configuration, Webpack tuning, or TypeScript errors in JSX templates. Every hour went into product features.
🌍
Mobile-first for Bahrain
The Bahrain food & beverage market is overwhelmingly mobile. Our 18KB PWA loads in under 1 second on 4G — faster than any React SPA could achieve without aggressive code-splitting and SSR.
🔀
Clear upgrade path
Moving to React or Next.js post-graduation is straightforward: the API contract, DB schema, and CSS design system are all framework-agnostic. The frontend is the only layer that would change.

Getting Started

Local Dev Setup

Get the full EchoEats stack running locally in under 10 minutes.

1
Clone & install
Clone the repo and install backend dependencies. The frontend needs no installation — just open any HTML file.
# Clone the repository
git clone https://github.com/echoeats/app.git
cd app

# Install backend dependencies
cd backend && npm install
2
Configure environment
Copy the example env file and fill in your local Postgres, Redis, and Stripe test keys.
cp .env.example .env

# .env values needed:
DATABASE_URL=postgresql://...
REDIS_URL=redis://localhost:6379
JWT_SECRET=your-secret-here
STRIPE_SECRET_KEY=sk_test_...
3
Run migrations & seed
Apply the Prisma schema to your local Postgres instance and seed demo data (vendors, listings, users).
# Apply schema migrations
npx prisma migrate dev

# Seed demo data
npx prisma db seed

# Open Prisma Studio (optional)
npx prisma studio
4
Start the servers
Run the Node API server, then open the frontend directly in your browser. The demo also works fully offline via localStorage without the backend running.
# Start the API server
npm run dev
# → http://localhost:3001

# Open frontend (no server needed)
open frontend/index.html
# → file:///path/to/index.html
